# Task 07 — 稽核與引用查核

## 目標
依 `docs/PLAN.md` §9 執行全面稽核,產出 `qa-report.md`（逐項 pass/fail + 修正清單）並套用修正；
若內容有改,重新渲染受影響的 PDF。

## 輸入
- `content/drafts/*.md`、`dist/*.pdf`
- `sources/source-map.md`、`sources/source-index.md`（§2）
- `content/knowledge-base.md`、`docs/GLOSSARY.md`
- `docs/PLAN.md` §9（稽核清單）、§3.8（準確性紅線）

## 不變式（AGENTS.md）
- 本階段是**全部不變式的強制執行關卡**。
- 不得在無證據下標記 pass；不得靜默刪除無法查證的宣稱（要嘛補來源,要嘛刪除並註明）。

## 步驟
1. **引用完整性**：逐份草稿掃描每個事實句 → 確認帶 `[S0X]`（大眾版查文末「依據」頁）→ 該編號能對應 `source-map.md` 並回溯到真實 §2 來源。列出所有孤兒宣稱。
2. **來源純度**：搜尋任何非 `security.apple.com` / `github.com/apple/*` 的來源或第三方轉述,應為零。
3. **無事實漂移**：挑共用事實（五大要求、transparency log、2026 控制權界線）在三份間比對,確認深淺不同但不矛盾。
4. **用詞一致**：對照 §4 GLOSSARY；確認大眾版正文無技術名詞洩漏。
5. **準確性紅線（§3.8）**：無「Apple 永遠看不到」式誇大；開發者版維持「宣稱 vs 可驗證」；S13 控制權界線正確；所有 `TODO(verify)` 已解或明確說明。
6. **渲染查核**：CJK 正常、註腳正確、分頁乾淨、三主題明確區隔、字型嵌入、頁數落在 §5 目標且非灌水。
7. **逐版驗收**（§9 逐版）：
   - 開發者:VRE 步驟可實際照做；達 28–40 頁且非灌水。
   - 使用者:類比不誤導；評估清單可用；達 14–22 頁。
   - 大眾:零術語；比喻不誤導；每頁文字克制；達 12–18 頁。
8. 產出 `qa-report.md`（逐項 pass/fail + 修正清單）；套用修正（或開 issue 追蹤）；內容有改則重渲染。
