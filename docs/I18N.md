# Bilingual Governance (I18N) — PCC Privacy Explainers

> 雙語發布治理：本 repo 如何維護繁體中文與英文兩種版本，並確保英文版**不越過來源邊界**。
> 從屬於 [`AGENTS.md`](../AGENTS.md)、[`docs/KNOWLEDGE_GOVERNANCE.md`](KNOWLEDGE_GOVERNANCE.md) 與 [`DESIGN.md`](../DESIGN.md) §10.4。
> 術語對照見 [`docs/BILINGUAL_TERMS.md`](BILINGUAL_TERMS.md)。

---

The English version is a source-aligned translation. It must not introduce claims that are absent from the Traditional Chinese source text or unsupported by the existing source map.

英文版是與繁中來源對齊的翻譯版本，不得新增繁中原文沒有、或既有 source map 不支持的技術 claim。

---

## 1. Source-of-record 與翻譯關係

- **繁體中文是 source-of-record**（權威記錄）。所有技術事實先在繁中定義（最終仍以 `content/knowledge-base.md` 與 `sources/source-map.md` 為準）。
- **英文版是 source-aligned translation**（與來源對齊的翻譯），不是獨立的內容來源。
- 檔案位置：繁中 `content/drafts/{01-developer,02-power-user,03-general-public}.md`；英文 `content/drafts/en/{developer,ai-user,general}.md`。
- 繁中檔案**保留原位置**，英文新增於 `content/drafts/en/`，不大幅搬移繁中以免破壞既有 build。

## 2. 英文版的硬規則

1. **不得新增繁中沒有的 technical claim。** 找不到對應繁中句與 `[S0X]` 的英文句，就是越界。
2. **不得弱化 limitation / uncertainty / source boundary。**「保證適用於 PCC 請求」「Apple 宣稱 vs 可獨立驗證」「無 reproducible builds」「威脅模型有假設邊界」「2026 擴展 ≠ 交第三方任意處理」等限定語，英文版必須保留等效語氣。
3. **不得把推論寫成事實。** 用 "the official documentation indicates"、"what can be confirmed"、"cannot directly infer"，不要寫成 "Apple proves" 或 "guarantees absolute privacy"。
4. **source reference 必須與繁中對齊。** 每個 `[S0X]` 在英文版出現於對應位置；數量與對應關係一致（PDF 文末註腳數應與繁中相同）。
5. **章節結構盡量對齊。** 英文版的標題、章節順序、callout 類型與繁中一一對應。
6. **英文可為自然語序調整句子，但不得改變 claim。** 可拆句、可調整語序以利英文閱讀，命題集合不可變。
7. **technical term 第一次出現保留英文原文與縮寫**（如 Private Cloud Compute（PCC）、attestation、Secure Enclave（SEP））；見 `BILINGUAL_TERMS.md`。
8. **若繁中內容更新，英文版需同步檢查**（見 §4 同步流程）。
9. **若英文因自然語氣新增句子，須確認該句只是說明語氣，不是新增 claim。** 語氣句不得攜帶新的事實、數字、能力或保證。

## 3. Translation pass 流程（怎麼翻而不新增 claim）

1. 先讀繁中對應檔與其 `[S0X]`，必要時回 `content/knowledge-base.md` / `source-map.md` 確認該句的來源範圍。
2. 逐句翻譯，**保留每個 `[S0X]` 與所有限定語**；callout 標記（`> [!TYPE] label`）原樣對應，label 用自然英文。
3. 技術詞首次出現用英文原文＋縮寫；大眾版（general）正文維持**無術語**（plain English 對應比喻）。
4. 不確定的詞或數字 → 不臆造；維持繁中既有的 `TODO(verify)` 標記，不在英文版「補滿」。
5. 圖片路徑：英文檔在 `content/drafts/en/`（比繁中深一層），圖片用 `../../../assets/...`。
6. 翻完做**命題比對**：英文句集合 ⊆ 繁中句集合（呈現可變、事實不可增）。
7. 跑 `npm --prefix build run qa`（check-refs 會掃 `content/` 含 `en/`，所有 `[S0X]` 須可追溯）。
8. 重建英文 Web（`build-html.mjs`）與英文 PDF（`build.mjs` 的 `*-en` audiences），目視抽查。

## 4. 同步與漂移控制

- 繁中更新（修事實、補來源、改限定語）→ **必須**在同一次工作檢查英文對應檔，避免雙語漂移。
- 英文 PDF 文末註腳數應等於對應繁中（目前 developer 16、ai-user 10、general 0）。數量不符即代表 `[S0X]` 對齊出問題。
- QA report（`qa-report.md`）的「Bilingual QA」區塊記錄每次雙語檢查結果。

## 4b. 雙語圖表（diagrams / illustrations）

- 繁中圖在 `assets/diagrams/*.svg`、`assets/illustrations/*.svg`；**英文圖在 `…/en/`** 子目錄。
- 英文圖由 **`build/i18n-svg.mjs`**（`npm --prefix build run i18n:svg`）從繁中圖＋文字對照表生成：**只翻譯 `<text>` 內容與換 Latin 字型，不改幾何／顏色**——等同圖的 source-aligned translation，不新增 claim。
- 繁中圖更新時，更新 `i18n-svg.mjs` 的對照表後重跑即可同步英文圖。對照表若有未涵蓋的中文，腳本會 **exit 1**（防漏譯）。
- 英文 drafts 以 `…/en/<name>.svg` 引用英文圖；繁中 drafts 用原圖。圖表文字術語對照同 [`BILINGUAL_TERMS.md`](BILINGUAL_TERMS.md)。

## 5. 與 AFM3 的關係

- AFM3（規劃中的姊妹 repo）可沿用本雙語結構與術語治理**思路**，但**各自維護自己的繁中 source-of-record 與英文翻譯**；不跨 repo 共用 claim 或 source map（見 `KNOWLEDGE_GOVERNANCE.md` §7）。

## 6. 不複製聲明

本 repo 借鑑 agent-friendly、Markdown-first knowledge base 的知識治理思路，但不複製 OpenKnowledge 的程式碼或實作。目的在於讓人類與 AI agent 都能更安全地查核、導覽、翻譯與更新文件。
